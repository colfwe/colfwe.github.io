1.1. 原因

为什么要脱离VisualStudio?

答: 用不惯，VisualStudio是用C#写的界面，挺卡的。

1.2. 做法

1.2.1. 安装MSVC编译器(即cl.exe)

从这里下载MSVC编译器: VSBT包含了一堆开发环境与编译器

1.2.2. 在cmd设置cl.exe的环境变量

因为Windows典型的销售产品策略

默认msvc的编译链接器是cl.exe，只能通过VS这个IDE上的CLI来启动，就是这么无语

但是如果想在命令行(也就是非VS编辑器上)上，使用msvc，那就遵循下面的步骤或 看文档

首先，这个东西是强依赖自己的机器配置的。所以如果在一台新的Windows机上复制粘贴下面的字符串是没卵用的。

在安装VSBT 2022版之后，应该在 C:\Program Files (x86)\Microsoft Visual Studio\2022\BuildTools\VC\Auxiliary\Build\

能看到有 vcvars32.bat 与 vcvars64.bat 和 vcvarsall.bat 这几个给Windows cmd运行的脚本文件。

序号 批处理脚本 作用
1 vcvars32.bat 将会在cmd当前会话下，可用cl.exe来生成x86的32位的产物。
2 vcvars64.bat 将会在cmd当前会话下，可用cl.exe来生成x86的64位的产物。
3 vcvarsall.bat 没试过，但应该能可用cl.exe来生成x86的32位与64位的产物。(但看名字感觉是)

所以我个人的机器上，要给cmd执行的可执行脚本文件，是这样的

请不要复制粘贴我自己的，原因是这个东西是强依赖自己的机器配置的 

%comspec% /k "C:\Program Files (x86)\Microsoft Visual Studio\2022\BuildTools\VC\Auxiliary\Build\vcvars64.bat"

在执行了后，等一下会提示说当前执行的脚本，将会产物什么位的产物，接下来就可以在cmd下用 

cl.exe /?

来查看cl.exe该编译链接器的用法了。

接下来，就在当前会话下，可以使用 cl.exe 了 

cl.exe 114514.c

1.2.3. 答疑解惑

  • 更多请看详细文档

    详细文档

  • 什么是 MSBuild

    相当于Windows下的makefile。Linux下Makefile/GNUMakefile，那么相对的，在Win32平台上自然就有它对应的"makefile"了。称为MSBuild。cmake的跨平台性是通过MSBuild来在Win32来构建源码的。

  • 嗯，接下来配置CMake

    文件结构 

    ./clcpp
   ./src
   CMakeLists.txt

    CMakeLists.txt单文件的个人配置 

    cmake_minimum_required(VERSION 3.30.0)
set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
set(CMAKE_BUILD_TYPE Debug)

project(clcpp LANGUAGES CXX)

if (WIN32)
   add_definitions(-DNOMINMAX -DUSE_MATH_DEFINES)
endif()

include_directories(
  ${clcpp_SOURCE_DIR}/src
)

file(GLOB sources CONFIGURE_DEPENDS
  "${clcpp_SOURCE_DIR}/src/hello.cpp"
)

add_executable(main ${sources})
add_custom_target(run COMMAND $<TARGET_FILE:main>)

    运行与使用 

    # 在当前cmd会话下，运行
%comspec% /k "C:\Program Files (x86)\Microsoft Visual Studio\2022\BuildTools\VC\Auxiliary\Build\vcvars64.bat"

# 配置cmake
cmake -B build

# cmake利用MSBuild来在Win32下构建产物
cmake --build build

# 运行(需要CMakeLists.txt写一个链接)链接
cmake --build build --targer run

# 接下来应该就能看到输出了。
